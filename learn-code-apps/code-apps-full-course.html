Lesson 1 · The code app mental model: what runs where

Win: draw the runtime on a whiteboard and say, without notes, what the platform owns and what your code owns.

Every conversation you'll lead — an executive asking "why are we paying for this?", an engineer asking "where does my code run?", a security reviewer asking "who serves the bundle?" — starts from this picture. Everything else in this course is a consequence of it.

Your repo (you own this)React/TypeScript source · Vite build · your Git, PR review, CI, security scanning — unchanged.
⬇ pac code push — uploads the compiled bundle only, one-way; the platform never stores source
Power Apps host (platform owns this)Serves your bundle · signs users in with Entra ID · enforces DLP, Conditional Access, sharing, CSP · meters usage.
⬇ user opens the app · ⬆ getContext() handshake via the @microsoft/power-apps SDK
User's browserRuns your SPA. All of your code executes here — there is no server tier of yours.
Connector layerEvery data call goes through a connector, metered as Power Platform Requests.
DataDataverse · Azure SQL · SharePoint · your own APIs behind custom connectors.

Dashed boxes: yours. Solid boxes: the platform's. The whole adoption trade lives in that dashed/solid split.

Five facts that anchor everything

1Your code runs only in the user's browser. A code app is a client-side SPA; anything that can't live in a browser must move behind a connector. [architecture]
2The host is the value proposition. Hosting, Entra sign-in, governance, delivery — that's what you stop building versus React + App Service. [overview]
3Data is connector-only. No connection strings, no direct DB. Typed clients are generated per data source; queries inherit delegation semantics, not SQL semantics. [connect-to-data]
4Git is the only home your source has. Push is one-way and compiled-only; there is no download. Lose the repo, lose the app. [cli]
5Every end user needs Power Apps Premium. No standard-connector discount, no M365 seeded coverage. Our tier works because we already hold Premium broadly. [prerequisites]
ADR tie-in. Fact 1 is why the ADR has a logic-placement policy at all: with no server tier of ours inside the platform, real business logic either degrades into flows/plugins or moves to our own Azure services behind custom connectors. The ADR chose the latter as the default.

Lesson 2 · The four app models — and the "why not canvas?" answer

Win: place any proposed app into the right Power Platform model in under a minute, and answer "why not canvas?" without hesitation.

The "why not canvas?" stakeholder answer

Someone senior will ask why your pro-code teams don't just use canvas apps — they look cheaper and the makers already know them. The honest answer has three parts:

  1. Toolchain. Canvas apps put the source of truth inside the platform, edited in a browser studio with Power Fx. Code apps keep it in Git with your PR review, your SAST/SCA scanning, your test frameworks. For a team that already runs those disciplines, canvas is a regression; code apps are continuity.
  2. Ceiling. Canvas UI is a control palette; complex interfaces fight it. A code app is your own React tree — the npm ecosystem included (within CSP limits, lesson 6).
  3. Honest cost caveat. Canvas apps using only standard connectors can run on cheaper licensing; code apps are Premium unconditionally. If a proposed app is a trivial form over SharePoint for users without Premium, canvas may genuinely be the right call — the tier is a default for pro-code teams, not a ban on canvas.
ADR tie-in. This is why the ADR is a tier, not a platform mandate: each model keeps its lane. Code apps take the pro-code, browser-based, connector-friendly lane; nothing in the ADR forbids a maker team from shipping a canvas form.

Lesson 3 · Licensing — who pays what

Win: answer any seat, price, or quota question in a meeting — including the 5,000-user math — without opening a document.

The three rules

  1. Every end user needs Power Apps Premium. The docs name it specifically and unconditionally; unlike canvas apps there is no cheaper "standard connectors only" path. [prerequisites]
  2. Microsoft 365 seeded rights do not cover code apps. The M365 grant is standard connectors + Dataverse for Teams only. No M365/O365 SKU by itself entitles a user to run a code app. [Licensing Guide]
  3. Developers build free. The Developer Plan grants up to 3 personal Dataverse environments; no license is needed to build or test. [dev plan]

Why quotas are a licensing topic

Each license carries a per-user daily Power Platform Request quota — Premium gets 40,000; per-app, PAYG, and M365-seeded get 6,000, non-poolable. A data-heavy app can exhaust 6,000 requests surprisingly fast, which is a real argument for Premium beyond the unlimited-apps entitlement.

ADR tie-in. Our tier's economics rest on one org-specific fact: target users already hold Premium broadly, so marginal license cost ≈ $0. That is why the ADR lists an EA change as a revisit trigger — the engineering trade-offs would survive a licensing change; the economics would not.

Lesson 4 · The connector data tier — where SQL intuition breaks

Win: spot the delegation trap and the quota ceiling in a design review before they ship.

How data access actually works

pac code add-data-source registers a connector against your app and generates typed TypeScript model + service files you call from your code. Supported sources include Dataverse, Azure SQL, SharePoint, and your own APIs via custom connectors. Two workflow facts pro-code developers find jarring: a connection schema change requires deleting and re-adding the data source to regenerate the types, and there is no path to call an arbitrary endpoint outside the connector model — the CSP blocks direct external calls (lesson 6). [connect-to-data]

The traps, in descending order of pain

1Silent truncation. Non-delegable queries evaluate client-side over a partial download: past 500 rows (configurable to 2,000 max) the rest are silently dropped — no error, no warning, wrong answer. Aggregates cap at 50,000 rows. This is the single most common wrong-result bug a reviewer should hunt for.
2No transactions or batch. There is no documented multi-row transaction or batch API. A "save these 30 rows atomically" requirement means server-side logic behind a custom connector — not a loop of single-row calls.
3Metering and ceilings. Every call is a Power Platform Request against the per-user daily quota (lesson 3). Synchronous flow calls time out at 120 seconds; gateway payloads cap around 3 MB. High-throughput or long-running work belongs in your own Azure backend, reached through a connector. [limits]
4Dataverse feature gaps. No FetchXML, no alternate keys, no polymorphic lookups from code apps. [dataverse]

The reviewer's patterns

Healthy designs push work to the server side of the connector: filtered views or stored procedures in SQL, Dataverse views, purpose-built endpoints on your own APIs. The client should receive result sets that are already small. When you see client-side filtering over a broad table pull in a design, that's the delegation trap wearing a disguise.

ADR tie-in. Admission criterion 2 ("data fits the connector model, queries survive delegation") exists because of this lesson. In review, operationalize it as: show me every query that could return more than 500 rows, and what happens to it.

Lesson 5 · ALM — one-way Git, solutions, and pipelines

Win: explain to a DevOps lead exactly which of our practices survive unchanged, which change shape, and which questions are still open.

Survives unchanged

The app is a normal React/TS repo. Git stays the source of truth; PR review, branch policy, SAST/SCA/SBOM scanning, unit tests (Vitest/Jest) and E2E (Playwright — the official templates ship with it) all run exactly as today. This is the genuine ALM advantage over canvas apps, whose source lives inside the platform.

Changes shape

Deployment artifactpac code push uploads the compiled bundle, one-way. There is no download command and Power Platform Git integration + Solution Packager are explicitly unsupported for code apps. The platform is a deployment target, never a source. [cli]
Transport unitApps attach to a solution at push (--solutionName). Solutions move between environments; connection references (CLI ≥ 1.51.1) and environment variables make one artifact target dev data in dev and prod data in prod. [push-to-solution]
First-party CDSolutions + Power Platform Pipelines is the documented dev→test→prod path. Azure DevOps / GitHub Actions can drive solution import/export instead, but for code apps specifically that route is community-documented, not first-party.
Environment prerequisiteCode app operations are disabled by default per environment — an admin toggle enables them. We confirmed this empirically: our spike's push returned 403 CodeAppOperationNotAllowedInEnvironment on an untouched environment.
ToolchainThe pac code command group is preview-and-deprecating; its npm replacement (power-apps init/run/push) is itself preview. Pin versions; platform engineering tracks the migration.

Still open — and honestly so

Whether a solution export carries the compiled bundle (so a downstream import reproduces the app) or every environment needs its own pac code push is undocumented. Our spike to answer it was halted at the environment toggle and consciously skipped; the ADR records the existing-CI default as provisional until this is answered. When a team asks "so how exactly does prod deployment work?", the correct current answer is: solution-based, existing-CI by default, one open question we will answer empirically before the paved road ships.

ADR tie-in. Point 4 (spike-gated CI/CD) and point 5 (platform engineering owns the paved road, including the environment toggle) both came from this lesson's material. Note the honest status marker inside the ADR itself — a decision document that hides its open questions is a liability.

Lesson 6 · Governance & security — what you inherit, what you owe

Win: walk a security reviewer through the inherited controls and the two exposures they'll care about, before they find them.

Inherited for free (the platform's half of the bargain)

Connector DLP policies are enforced against code apps at launch and at runtime; per-app Conditional Access, tenant isolation, sharing limits, App Quarantine, and admin-center health metrics all apply. For a self-hosted app, each of these is something your org would have had to build or buy; here they arrive with the host. This inherited-governance story is the strongest security argument for the tier.

The locked-down CSP (feature and friction)

The default Content Security Policy is restrictive: connect-src 'none' blocks every external call — including Application Insights telemetry — until an environment admin allow-lists the endpoint; frame-ancestors restricts embedding to *.powerapps.com; external CDN scripts are blocked; worker-src 'none' forecloses service workers (and with them offline/PWA — lesson 7). An npm package that phones home simply breaks. Plan the allow-list (telemetry first) into the environment baseline, not as a per-app surprise. [system limits]

The two exposures to disclose proactively

  1. The bundle endpoint is public. Compiled assets are served from an endpoint with no IP restriction. Mitigation: Conditional Access by location — and an absolute rule that no secrets or sensitive data ship in the bundle. Everything privileged goes through connectors, which authenticate per user.
  2. Auth has no escape hatch. The host performs Entra ID sign-in; you cannot bring another IdP or a custom login flow. Backend extension happens only via custom connectors (Entra OAuth or managed identity). For internal-only apps this is a simplification; state it as a constraint, not a surprise.

One inventory caveat: the CoE Starter Kit is no longer maintained — governance inventory now lives in the admin center — and whether code apps surface there as a distinct app type is unconfirmed. Your admins should verify before promising reporting granularity.

ADR tie-in. The paved road (point 5) exists mostly because of this lesson: CSP allow-listing, DLP coordination, and Conditional Access are environment-level concerns no app team can self-serve. An app that ships before the telemetry endpoint is allow-listed flies blind.

Lesson 7 · Limits & saying no — the tier-admission checklist

Win: run a tier-admission review in fifteen minutes and defend every "no" with a category, not an opinion.

Classify before you argue

Every limitation belongs to one of three categories, and the category decides how you talk about it:

(a) Documented hard limitTrue today, could change: app size limits, Excel Online connector unsupported, no FetchXML from Dataverse, Test Studio/Test Engine coverage absent.
(b) Preview gapTooling churn likely to close: the pac→npm CLI migration, Dataverse image/file upload in preview, CLI-created connections in preview.
(c) Inherent architectural constraintWill not change without a redesign of the model: no server tier of yours, connector-only data, no mobile player (Microsoft: "in our backlog… no current ETA or plan"), no offline/PWA (CSP worker-src 'none'), Entra-only auth.

Category (c) items are the ones that rule apps out of the tier permanently; category (b) items are why the ADR carries a 12-month criticality carve-out. Never present a (b) as a (c) or vice versa — stakeholders remember. [system limits], [official repo]

The admission checklist, operationalized

  1. Audience & surface: internal, desktop-browser. Any mobile, offline, or PWA requirement → out (category c). Teams/SharePoint embedding needs a frame-ancestors exception — check early.
  2. Data shape: every query that could exceed 500 rows must be delegable or server-filtered. Bulk writes or multi-row transactions → own backend behind a custom connector, or out.
  3. Volume: requests/user/day within quota with margin; nothing synchronous beyond 120s; payloads under ~3 MB.
  4. Auth: Entra-only, internal users. External identities → out.
  5. Deployment: team accepts solution-based transport and the paved road.
  6. Criticality: tier-1/business-critical → wait out the carve-out.

The exit story (asymmetric, by design)

If the company ever leaves: the React/Vite front-end ports cleanly — it's standard code. What gets rewritten: every SDK call and generated connector service, plus auth and hosting. What survives untouched: logic already externalized into your own Azure services behind custom connectors. That asymmetry is why the ADR's logic-placement default is the single highest-leverage policy in the tier — it converts unbounded lock-in into a bounded, estimable rewrite.

ADR tie-in. This lesson is ADR points 1–3 in operational form. When you reject an app, cite the criterion and the category: "criterion 1, category (c) — offline is architecturally unavailable" ends arguments; "it doesn't feel like a fit" starts them.

Lesson 8 · Capstone — defending the ADR

Win: survive six realistic stakeholder challenges. Answer each aloud before expanding the model answer.

The executive: "We already pay for Azure. Why add a platform?"

We're not adding spend — our users already hold Power Apps Premium, so the tier's marginal license cost is approximately zero. What we're removing is build cost: hosting, Entra sign-in, sharing, DLP, and Conditional Access arrive from the host instead of being built per app. The ADR binds this to a condition: if our EA changes and Premium coverage lapses, the decision re-opens — that's a named revisit trigger.

The engineer: "Why can't I just query the database?"

There's no path to it — data access is connector-only, by architecture. And the connector tier has different semantics than SQL: non-delegable queries silently truncate at 500 rows, there are no multi-row transactions, and every call is metered. Design server-side: filtered views, stored procedures, or our own API behind a custom connector. If your app genuinely needs direct database semantics, it fails admission criterion 2 and belongs in the pro-code lane — that's the tier working as designed, not a workaround.

The security reviewer: "Who serves your bundle, and from where?"

The Power Apps host, from a public endpoint with no IP restriction — mitigated by Conditional Access by location, and by policy nothing sensitive ships in the bundle; all privileged access is per-user through connectors. In exchange we inherit runtime DLP, per-app Conditional Access, tenant isolation, and App Quarantine. Source never leaves our Git, so your SAST/SCA pipeline covers it unchanged.

The finance partner: "What does 5,000 users actually cost?"

At list, $100k/month; at the documented 2,000+ volume tier, $60k/month; per-app pay-as-you-go would be at most $50k/month for a single app and scales with active use — but its validity for code apps is unconfirmed and we don't rely on it. For us the practical number is ~$0 marginal, because target users already hold Premium. The exposed assumption is the EA — hence the revisit trigger.

The PM: "Can we build the field-technician app on this?"

No — and this is a permanent no, not a timing no. Code apps don't run in the mobile player (Microsoft: backlog, no ETA or plan) and the CSP blocks service workers, so offline is architecturally unavailable. That app takes the pro-code lane. Admission criterion 1, category (c).

The architect: "Why not just use Power Platform Pipelines like the docs say?"

Default is our existing Azure DevOps/GitHub CI at the solution layer, because one CD system with our PR gates beats running a second deployment stack. That default is explicitly provisional: whether a solution export carries the compiled bundle is undocumented, our spike to answer it was halted at the environment toggle, and the ADR records Pipelines as the fallback if the answer forces it. We answer it empirically before the paved road ships.
